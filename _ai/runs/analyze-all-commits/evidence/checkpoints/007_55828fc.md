# Checkpoint 007: 55828fc

## Commit 标题
Modify：对整个引擎只有一个的类，用静态函数，而不是instance()->create()

## 核心改动
1. **单例模式重构**：
   - 修改 `Runtime/Managers/Manager.h` 基类，调整9行代码，将原来的instance()实例获取方式改为更简洁的静态函数接口，避免每次调用都要写instance()->xxx的冗余代码
   - 重构所有管理器类的接口设计，将全局唯一的管理器类的公共方法改为静态函数，对外提供更简洁的调用方式，内部依然保持单例实现，用户不需要关心实例获取细节

2. **各管理器适配**：
   - 修改 `Runtime/Managers/SceneManager.cpp/h` 调整8行代码，将场景管理器的接口改为静态函数方式
   - 修改 `Runtime/Managers/WindowManager.cpp/h` 调整24行代码，重构窗口管理器的接口设计，改为静态函数调用方式
   - 修改 `Runtime/Platforms/Windows/WindowManagerWindows.cpp/h` 调整88行代码，适配基类接口变更，简化内部实现
   
3. **核心框架适配**：
   - 修改 `Runtime/Core/Game.cpp` 调整4行代码，适配新的管理器调用方式，简化Game类中对各个管理器的调用代码

## 涉及模块
- 管理器架构
- 场景管理
- 窗口管理
- 引擎核心框架

## 风险点
- 静态函数接口设计需要考虑线程安全性，多线程环境下调用可能存在问题
- 重构涉及多个模块，可能存在遗漏修改的地方，导致部分功能无法正常工作
- 静态函数的可测试性比实例方法差，后续单元测试可能需要额外处理
- 这种设计模式限制了管理器的扩展性，如果后续需要多个实例会比较困难

## 推测错误原因
原来的单例调用方式比较繁琐，每次都要先获取实例再调用方法，代码冗余且不够直观。通过将全局唯一的管理器类的公共接口改为静态函数，可以简化调用方式，使代码更简洁易读，同时内部依然保持单例模式的优势，保证全局唯一实例和资源管理。这种调整主要是API层面的优化，提升开发体验和代码可读性。
