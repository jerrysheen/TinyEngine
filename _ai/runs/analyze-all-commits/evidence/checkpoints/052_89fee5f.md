## 核心修改
### 1. Resize 从「立即调用」改为「渲染指令 + 帧尾执行」
- 删除 `RenderAPI::OnResize(int,int)` 虚接口；窗口尺寸变更改为 **`RenderAPIWindowResize(Payload_WindowResize)`**，由具体 D3D12 实现完成原先 `OnResize` 中的 swapchain/RTV/DSV/视口/裁剪矩形逻辑（代码体基本平移）。
- `RenderCommand` 增加 **`RenderOp::kWindowResize`** 与 **`Payload_WindowResize { width, height }`**；`CommandData` 联合体增加 `onWindowResize` 字段。

### 2. 投递与消费路径
- `RenderEngine::OnResize` 不再直接调 `RenderAPI::OnResize`，改为 **`Renderer::ResizeWindow`**。
- `Renderer::ResizeWindow` 构造 `DrawCommand`（`kWindowResize`），**`PushBlocking`** 进渲染线程队列，保证顺序。
- `Renderer::RenderLoop` 内：遇到 `kWindowResize` 时暂存到 `pendingResize` / `hasResize`；在 **`kEndFrame` 且已执行 `RenderAPIEndFrame()` 之后** 再调用 `RenderAPIWindowResize`，避免在帧中间改 swapchain/资源与命令列表状态冲突。

### 3. 其它
- `Culling.cpp` 增加一行 todo 注释（bounds/快速剔除）。

## 风险点
1. **多重重置**：若同一帧内多次 resize，仅保留最后一次 `pendingResize` 是否足够需结合窗口消息合并策略验证。
2. **与 UI/ImGui 尺寸**：帧尾才更新 viewport/scissor，主线程与渲染线程对「当前分辨率」的感知可能差一帧。
3. Blocking push：resize 路径阻塞提交线程，极端频繁 resize 时可能卡顿。

## 可能的错误源
- 在 `BeginFrame`/`Draw` 中途执行 `ResizeBuffers` 与 GPU 使用中的 backbuffer 冲突；本提交将 resize 推迟到 **EndFrame 之后** 缓解。
- 移除 `OnResize` 后若有代码仍直接调用需全部改为 `Renderer::ResizeWindow` 或等价路径。
