# 已知技术债（待处理）

1. D3D12 绑定开销
说明：`Runtime/Platforms/D3D12/D3D12RenderAPI.cpp` 的 `RenderAPISetMaterial` 每次都会重复绑定全局 SRV（AllObjectData / AllMaterialData / Visibility），即使 RootSignature 未切换也会重复执行。
影响：CPU 端绑定与 descriptor 复制开销偏大。
建议：将全局 SRV 绑定移到 `RenderAPISetRenderState`（RootSig 变更时）或 `RenderAPIBeginFrame`，并在 `RenderAPISetMaterial` 做材质缓存，减少重复绑定。

• 主要风险

  1. 帧边界还没有真正收口，RenderPipeline 仍在控制 frame submit，后续任何 GPUScene 接入都会继续污染职责边界。
     [LagacyRenderPipeline.cpp](/D:/GitHubST/TinyEngine/Runtime/Renderer/RenderPath/LagacyRenderPipeline.cpp#L19) 到
     #L23 在 Prepare 阶段发 BeginFrame/SetFrame，[LagacyRenderPipeline.cpp](/D:/GitHubST/TinyEngine/Runtime/Renderer/
     RenderPath/LagacyRenderPipeline.cpp#L53) 又直接 EndFrame。这会导致 RenderEngine 虽然名义上有 PrepareFrame/
     BuildFrame，但实际不能完整代表“一帧”的生命周期，后面想加 SubmitFrame、批量 cmd stream、frame profiling 都会被卡
     住。
  2. UploadPagePool 的 ticket 生命周期没有接到主链上，存在上传页不回收或回收时机失真的风险。[UploadPagePool.cpp](/D:/
     GitHubST/TinyEngine/Runtime/Renderer/UploadPagePool.cpp#L47) 的 OnSubmitted() 和 #L61 的 Recycle() 目前没有进入正
     常帧流程，但 [GPUScene.cpp](/D:/GitHubST/TinyEngine/Runtime/Scene/GPUScene.cpp#L107)、#L230、#L255、#L280 已经大
     量依赖这个 pool。现在能跑不代表安全，场景一大就容易变成 upload heap 持续增长或旧页复用不确定。
  3. FrameTicket 比较语义有 bug，会直接影响 upload page 回收。[FrameTicket.h](/D:/GitHubST/TinyEngine/Runtime/
     Renderer/FrameTicket.h#L31) 的 operator<= 实现成了 <，而 [UploadPagePool.cpp](/D:/GitHubST/TinyEngine/Runtime/
     Renderer/UploadPagePool.cpp#L63) 又依赖这个比较做 recycle。等值 fence 不会被回收，这种问题短期不一定炸，但会积累
     成隐性内存/页池滞留。
  4. GPUScene 现在“数据同步”基本接进来了，但“渲染执行路径”还没恢复，处于半成品状态。[RenderEngine.cpp](/D:/GitHubST/
     TinyEngine/Runtime/Renderer/RenderEngine.cpp#L51) 到 #L57 已经在每帧调用 mGPUScene.Update、消费 dirty、上传 copy
     op；但 [GPUSceneRenderPipeline.cpp](/D:/GitHubST/TinyEngine/Runtime/Renderer/RenderPath/
     GPUSceneRenderPipeline.cpp#L7) 整个 RecordAndFlush 仍是注释掉的。也就是说你已经承担了同步成本，但还没拿到
     GPUScene 路径的收益，调试面会扩大。
  5. RenderEngine::EndFrame() 只清 CPUScene，没有成为真正的帧尾收敛点。见 [RenderEngine.cpp](/D:/GitHubST/TinyEngine/
     Runtime/Renderer/RenderEngine.cpp#L99)。如果后面继续把 upload recycle、GPUScene frame-local reset、统计采样分别塞
     到别处，Tick 重构会再次碎掉。
  6. GPUScene 的 copy/update 逻辑中有一些容易形成数据错写的细节。比如 [GPUScene.cpp](/D:/GitHubST/TinyEngine/Runtime/
     Scene/GPUScene.cpp#L265) 到 #L272 在创建 AABB 时上传的是 &mPerObjectDatas[renderID]，不是 AABB 数据本身，这很像直
     接写错源数据；如果确实如此，这是功能性 bug，不只是架构问题。

  短期内最值得做的优化

  1. 先把帧主链改成四段，别急着补全 GPU culling。
     建议收口成：PrepareFrame 只做 slot/ticket/dirty 消费，RecordFrame 只录制 copy+draw，SubmitFrame 只发 SetFrame +
     EndFrame，EndFrame 只清理 CPU/GPU frame-local 状态。这样后面不管是 Legacy 还是 GPUScene，都能挂在同一帧骨架上。
  2. 把 UploadPagePool 真正接上 FrameTicket。
     在复用 frame slot 前对旧 ticket 做 Recycle()，在 render thread 发布 submission 后或主线程 submit 封口后做
     OnSubmitted()。这是当前最有价值的短期修复，投入小，能立刻把 upload 生命周期做实。
  3. 修掉两个低成本高收益 bug。
     一个是 [FrameTicket.h](/D:/GitHubST/TinyEngine/Runtime/Renderer/FrameTicket.h#L31) 的比较符；一个是检查
     [GPUScene.cpp](/D:/GitHubST/TinyEngine/Runtime/Scene/GPUScene.cpp#L265) 的 AABB 上传源数据。前者是生命周期 bug，
     后者可能是实际渲染错误。
  4. 把 LagacyRenderPipeline 里的 frame 控制挪走。
     [LagacyRenderPipeline.cpp](/D:/GitHubST/TinyEngine/Runtime/Renderer/RenderPath/LagacyRenderPipeline.cpp#L19) 到
     #L23、#L53 这些应该移到 RenderEngine。Pipeline 只保留 Prepare pass data / Record pass commands。这是最关键的职责
     降噪，能直接降低后续改动风险。
  5. 给关键阶段加最少量 profiling 计数。
     建议先加三个：WaitForFrameAvailable 等待时长、mCurrentCopyOp->size()、UploadPagePool 当前 free/pending/inflight
     页数。现在已经有 profiler 宏，这些数据能非常快地告诉你瓶颈到底在 fence、copy，还是 page churn。
  6. 暂时不要恢复完整的 GPUSceneRenderPipeline，先做最小通路验证。
     先验证 SceneDelta -> CPUScene -> GPUScene buffer copy -> shader 读取 这条链是对的，再上 compute culling 和
     indirect。否则你会同时调“帧边界问题”和“GPUScene 功能问题”，排障成本会翻倍。
