# BugFix: 近三天引擎踩坑总结

这份文档只记录 `TinyEngine` 这三天内暴露出来的引擎侧坑点。  
重点不是流水账，而是：

- 当时犯了什么判断错误
- 引擎里真正漏掉了什么约束
- 以后 review 时应该优先看哪里

## 1. 把 `Present` 误当成 frame 资源可复用边界

### 现象

- 动态 descriptor allocator 原来会在 `Present` 后立即 reset
- 运行时很多时候没事
- RenderDoc replay 更容易出问题

### 当时的错误理解

- 以为“执行到 `Present`，这一帧就结束了”
- 于是直觉上认为这时候回收 per-frame descriptor 是安全的

### 真正的问题

- `Present` 只是 CPU 提交流程的一步
- 不代表 GPU 已经消费完这帧引用过的 descriptor
- 真正的复用依据应该是 fence / `FrameTicket`

### 以后 review 要看什么

- 任何 per-frame allocator / ring buffer / 临时 heap 的 reset
- 都不要只看“代码流程走到哪”
- 要看“GPU 完成点到底绑定在什么 fence 上”

## 2. 把 `ResetFrameAllocator()` 误理解成“清空 descriptor”

### 现象

- 讨论里一开始容易把问题想成“reset 后 descriptor 空了，中间有空泡期”

### 当时的错误理解

- 以为 reset 会把这一帧的 descriptor 清掉
- 于是把问题理解成“GPU 可能读到空表”

### 真正的问题

- 当前实现里 reset 只是把动态分配指针移回 frame 段起点
- descriptor 内容本身并没有被清掉
- 真正危险的是后续新的分配又覆盖了同一段槽位

### 以后 review 要看什么

- 看到 `Reset` / `Recycle` / `Clear`
- 先确认它到底是“真的清内容”
- 还是“只回卷分配指针”

## 3. 误以为 `LagacyRenderPipeline` 没有走动态 descriptor 路径

### 现象

- 讨论中一度认为 legacy 不走 GPUScene compute
- 所以可能也不会碰到同样的 descriptor 生命周期问题

### 当时的错误理解

- 把“没走 GPUScene”误推成了“没走动态 descriptor table”

### 真正的问题

- `LagacyRenderPipeline -> OpaqueRenderPass -> RenderPass::IssueRenderCommandCommon`
- 里面明确会调用 `SetMaterialData`
- `RenderAPISetMaterial` 会从 bindless heap 的动态区临时分配 descriptor table

### 以后 review 要看什么

- 不要只按“哪条渲染路径”做高层判断
- 要一路追到最终的 `RenderAPISetXXX`
- 看它到底是直接绑全局 heap，还是临时拷 descriptor table

## 4. 误以为“以前没崩”就说明这条链路基本安全

### 现象

- legacy 路径以前没有明显炸
- 原生运行很多时候也稳定

### 当时的错误理解

- 直觉上容易把“没出错”当成“逻辑没问题”

### 真正的问题

- 这类 bug 是典型的时序型未定义行为
- 很多时候只是因为新旧 descriptor 内容刚好接近，或者 GPU 时序比较幸运
- RenderDoc replay 更容易把这种问题放大

### 以后 review 要看什么

- 对 GPU 生命周期问题，不要用“目前跑得稳”作为安全证明
- 尤其是 descriptor / UAV / indirect / transient buffer 这几类资源

## 5. Compute root signature 的 descriptor range 生命周期写错了

### 现象

- compute root signature 创建逻辑里，把局部 `CD3DX12_DESCRIPTOR_RANGE` 的地址传给了 `InitAsDescriptorTable(...)`

### 当时的错误理解

- 默认认为 `InitAsDescriptorTable(...)` 会复制内容
- 没意识到这里只是把指针保存起来，后面序列化时还会再访问

### 真正的问题

- 局部变量提前析构
- 后面 `D3D12SerializeRootSignature(...)` 访问的是悬空指针

### 以后 review 要看什么

- 所有 `InitAsDescriptorTable(...)`
- 都要确认传进去的 range 生命周期至少覆盖到 root signature serialize 完成

## 6. Root signature hash 覆盖不完整，导致 shader 可能错误复用 root signature

### 现象

- 后面补了 `sampler + texture + rwTexture` 到 rootsig hash

### 当时的错误理解

- 一开始只按部分资源类型生成 key
- 默认认为“差不多的 shader 应该能共用 root signature”

### 真正的问题

- shader 绑定布局不同，但 hash 没体现出来
- 最后可能错误复用不兼容的 root signature

### 以后 review 要看什么

- 所有 shader reflection -> root signature key 的生成逻辑
- 必须覆盖：
  - CBV
  - SRV
  - UAV
  - Texture
  - RWTexture
  - Sampler

## 7. UAV 的问题不能只看 resource state，还要看前后顺序关系

### 现象

- 后面补了 `UAVBarrier`
- RenderDoc / GPUValidation 才更稳定

### 当时的错误理解

- 以为资源一直处于 `STATE_UNORDERED_ACCESS` 就够了

### 真正的问题

- state 只说明“现在能不能这么用”
- 不能说明“前一个 UAV 写的结果，后一个阶段什么时候可见”
- 真实有数据依赖时，仍然需要显式 `UAV barrier`

### 以后 review 要看什么

- 所有 `compute -> compute`
- `compute -> indirect`
- `compute -> srv`
- 如果前后真的有数据依赖，必须专门审 barrier

## 8. GPU validation 一开，隐式状态问题马上暴露

### 现象

- 开了 GPU validation 后，`FinalBlit` 前的隐式状态转换马上报错

### 当时的错误理解

- 平时跑得通时，容易默认资源状态“应该已经对了”

### 真正的问题

- 某些 pass 之间只是“碰巧没出事”
- 但没有显式表达资源用途切换
- validation 一开就会把这种隐式假设打掉

### 以后 review 要看什么

- 每个 pass 的输入输出资源
- 都应该明确写出状态切换依据
- 不要依赖“上一个 pass 大概已经切好了”

## 9. Editor GUI 自己的 command allocator / command list 也需要独立 fence

### 现象

- 后面专门给 GUI 增加了自己的 fence

### 当时的错误理解

- 默认认为 GUI 路径跟主渲染共用当前 frame 节奏就够了

### 真正的问题

- GUI 自己也会提交 command list
- 如果没有独立 fence，就可能在 allocator 尚未安全时提前 reset

### 以后 review 要看什么

- 任何“附加提交”的系统：
  - GUI
  - DebugDraw
  - Copy helper
  - Async path
- 只要自己提交命令，就要有自己的完成点或明确并入主 frame fence

## 10. 动态 descriptor 既然 allocate 在哪条线程，reset 也最好放同一 ownership 下

### 现象

- 后面把动态 descriptor reset 放到了 render thread 的 `RenderAPISetFrame()`

### 当时的错误理解

- 只按“语义上像 frame begin”来放 reset
- 容易忽略真正做分配的是哪条线程

### 真正的问题

- 如果主线程 reset、渲染线程 allocate
- ownership 会变乱
- 以后出问题时更难推断

### 以后 review 要看什么

- 线性分配器、临时表、frame scratch 一类资源
- 最好保证 allocate / reset / recycle 的线程归属清晰一致

## 11. 大纹理和 replay 压力会放大问题，但不是根因

### 现象

- 4K DDS、显存压力、RenderDoc capture 会明显放大异常

### 当时的错误理解

- 容易把“放大器”误判成“根因”

### 真正的问题

- 资源体量只是让边缘 bug 更容易被打出来
- 根因还是同步、状态、生命周期表达不完整

### 以后 review 要看什么

- 分清“这是直接 bug”
- 还是“这是让 bug 更容易浮出来的压力因素”

## 一句话版 review checklist

后面 review 渲染改动时，优先问这几句：

1. 这个资源什么时候才算真的可复用，是靠 `Present` 还是靠 fence？
2. 这个 `Reset/Clear/Recycle` 是真清内容，还是只回卷指针？
3. 这条渲染路径最终是直接绑全局 heap，还是临时拷 descriptor table？
4. root signature 的 key 是否完整覆盖了 shader 实际绑定布局？
5. 这里缺的是 resource state，还是缺的是 UAV 可见性/顺序 barrier？
6. 这块“现在没崩”，是不是只是因为时序幸运或内容恰好相同？
